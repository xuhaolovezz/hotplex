# HotPlex 双数据库支持（SQLite + PostgreSQL）技术规格

> **版本**: v1.0  
> **日期**: 2026-05-26  
> **状态**: Proposed  
> **Issue**: [#487](https://github.com/hrygo/hotplex/issues/487)
> **概述**: 在保持现有 SQLite 生产级支持的前提下，新增 PostgreSQL 作为可选数据库后端。通过 `dbutil.Dialect` 抽象层隔离 SQL 方言差异，零侵入改造现有 Store 接口。

---

## 1. 概述

### 1.1 动机

| 场景 | SQLite | PostgreSQL |
|------|--------|------------|
| 单机部署 / 嵌入式 | ✅ 天然优势 | ❌ 需要独立进程 |
| 高并发写入（50+ session 同时） | ⚠️ `WriteMu` 串行化，吞吐上限 | ✅ MVCC 天然并发 |
| 多实例共享数据 | ❌ 文件锁，单写者 | ✅ 原生支持 |
| 生产运维（备份/监控/告警） | 手动 | 成熟生态（pgBackRest, pg_stat_statements） |
| 跨区域只读副本 | ❌ | ✅ 流复制 |
| JSON 查询 | `json_extract()` 有限 | `jsonb` GIN 索引 + 丰富操作符 |

**核心诉求**：用户应能通过一个配置开关（`db.driver: postgres`）切换到 PostgreSQL，无需修改业务代码。

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **接口不变** | 现有 `session.Store`、`cron.Store`、`eventstore.EventStore` 接口零改动 |
| **SQLite 优先** | SQLite 仍是默认选项，PostgreSQL 是 opt-in |
| **零 ORM** | 保持 raw SQL + `database/sql`，不引入 GORM/xorm/sqlx |
| **Dialect 抽象** | 仅抽象最小必要差异（占位符、引用、错误码），非完整 Dialect 接口 |
| **构建标签可选** | CGO-free 默认使用 `modernc.org/sqlite`，可选 `mattn/go-sqlite3` |

### 1.3 非目标

- ❌ MySQL / MSSQL 支持（架构可扩展，但本次不实现）
- ❌ ORM 引入（保持 raw SQL）
- ❌ 数据库迁移工具（保持 goose）
- ❌ 读写分离 / 主从复制
- ❌ 连接池中间件（使用标准 `database/sql` 池）
- ❌ 分布式事务

## 3. SQL 方言差异矩阵

### 3.1 DDL 差异

| 特性 | SQLite | PostgreSQL | 影响范围 |
|------|--------|------------|----------|
| 自增主键 | `INTEGER PRIMARY KEY AUTOINCREMENT` | `BIGSERIAL PRIMARY KEY` / `GENERATED BY DEFAULT AS IDENTITY` | 所有迁移 |
| 布尔类型 | `INTEGER CHECK(x IN (0,1))` | `BOOLEAN` | 迁移 + boolToInt() |
| 时间戳类型 | `DATETIME` | `TIMESTAMP` / `TIMESTAMPTZ` | 迁移 |
| 浮点类型 | `REAL` | `NUMERIC` / `DOUBLE PRECISION` | turns 表 |
| 默认时间 | `datetime('now')` | `NOW()` | api_key_users 迁移 |
| ALTER TABLE | 无 `IF NOT EXISTS` | `ADD COLUMN IF NOT EXISTS` | 增量迁移 |
| CHECK 约束 | 语法一致 | 语法一致 | ✅ 无差异 |

### 3.2 DML 差异

| 特性 | SQLite | PostgreSQL | 影响范围 |
|------|--------|------------|----------|
| 占位符 | `?` | `$1`, `$2` | **所有查询** (需 Rebind) |
| 标识符引用 | `` `col` `` 或 `"col"` | `"col"` | CREATE TABLE / ALTER TABLE |
| UPSERT | `ON CONFLICT(id) DO UPDATE SET col=excluded.col` | `ON CONFLICT (id) DO UPDATE SET col=EXCLUDED.col` | sessions.upsert |
| RETURNING | 3.35+ 支持 | 原生支持 | 可选的 insert 优化 |
| 字符串拼接 | `\|\|` | `\|\|` | ✅ 一致 |
| COALESCE | 支持 | 支持 | ✅ 一致 |
| LIMIT/OFFSET | 支持 | 支持 | ✅ 一致 |

### 3.3 函数差异

| 功能 | SQLite | PostgreSQL | 影响范围 |
|------|--------|------------|----------|
| JSON 提取 | `json_extract(col, '$.key')` | `col->>'key'` / `col#>'{key}'` | cron 索引 005 |
| 当前时间 | `CURRENT_TIMESTAMP` | `NOW()` / `CURRENT_TIMESTAMP` | 默认值 |
| 布尔值 | `0` / `1` (整数) | `TRUE` / `FALSE` (原生) | boolToInt() |
| 唯一约束错误 | `"UNIQUE constraint failed"` (string match) | `pq.Error.Code == "23505"` | isSQLiteUnique() |

### 3.4 并发控制差异

| 特性 | SQLite | PostgreSQL |
|------|--------|------------|
| 写并发 | 单写者 (WriteMu 串行化) | MVCC 多写者 |
| 读并发 | WAL 模式支持 | MVCC 原生支持 |
| 行锁 | 不支持 | `SELECT ... FOR UPDATE` |
| 死锁 | 不存在 | 可能发生 (需 `IsDeadlock()`) |
| 连接池 | MaxOpenConns=1 (写), 读可 >1 | MaxOpenConns=25 (典型) |
| Busy 处理 | `busy_timeout` PRAGMA + WriteMu | `lock_timeout` 参数 |

### 4.3 `dbutil.Dialect` — 核心抽象

```go
// internal/dbutil/dialect.go

type Dialect string

const (
    DialectSQLite   Dialect = "sqlite"
    DialectPostgres Dialect = "postgres"
)

// Rebind 将 SQLite 风格的 ? 占位符转换为 PostgreSQL 的 $N 格式。
// 正确处理字符串字面量、转义引号、双引号标识符、-- 注释、/* */ 块注释、
// $$ 美元引用。SQLite dialect 下为恒等变换。
func (d Dialect) Rebind(query string) string

// Placeholder 返回第 n 个占位符字符串。
//   SQLite:    "?"
//   Postgres:  "$1", "$2", ...
func (d Dialect) Placeholder(n int) string

// QuoteIdent 用数据库原生的引用字符包裹标识符。
//   SQLite:    `col` → "col"
//   Postgres:  col → "col"
func (d Dialect) QuoteIdent(name string) string

// BoolValue 返回适合数据库的布尔值。
//   SQLite:    1 或 0
//   Postgres:  true 或 false
func (d Dialect) BoolValue(b bool) any

// IsUniqueViolation 判断给定 error 是否为唯一约束冲突。
//   SQLite:    匹配 "UNIQUE constraint failed"
//   Postgres:  检查 pq.Error.Code == "23505"
func (d Dialect) IsUniqueViolation(err error) bool
```

**关键设计决策**：

1. **Dialect 不是 interface，是 string 常量类型**：避免为每个方言创建实现 struct 的开销。所有方法都是纯函数，无状态。
2. **Rebind() 使用状态机**（参考 autobrr/qui 的生产级实现）：
   - 4 个状态：`Normal` → `SingleQuote` → `DoubleQuote` → `DollarQuote`
   - 正确处理 `'it''s'`, `"my""table"`, `$$dollar$quoted$$`, `-- comments`, `/* blocks */`
3. **不做完整 Dialect 接口**：Grafana 的 35 方法 Dialect 过度设计。HotPlex 只需要 5 个方法覆盖所有差异。

### 4.4 `dbutil.DB` — 连接包装器

```go
// internal/dbutil/db.go

type DB struct {
    *sql.DB
    dialect  Dialect
}

// Open 根据配置创建数据库连接。
func Open(cfg *config.DBConfig) (*DB, error) {
    dialect := ParseDialect(cfg.Driver)
    switch dialect {
    case DialectSQLite:
        db, err := sql.Open("sqlite", cfg.SQLite.DSN())
        // 应用 PRAGMA (沿用现有 InitSQLiteDB)
        // SetMaxOpenConns(1 + readers)
        return &DB{DB: db, dialect: dialect}, nil
    case DialectPostgres:
        db, err := sql.Open("postgres", cfg.Postgres.DSN())
        db.SetMaxOpenConns(25)
        db.SetMaxIdleConns(5)
        db.SetConnMaxLifetime(5 * time.Minute)
        return &DB{DB: db, dialect: dialect}, nil
    }
}
```

**关键设计决策**：

1. **嵌入 `*sql.DB`**：允许 `DB` 实例直接用于所有现有 `database/sql` 调用（`ExecContext`, `QueryRowContext` 等）。Store 层代码几乎不需要改动。
2. **不实现 `BeginTx`**（Phase 1）：`BeginTx` 涉及 `WriteMu` 管理，仅在 eventstore 使用。PG 下直接委托给 `*sql.DB.BeginTx`。
3. **PG 连接池独立管理**：PG 每个 Store 获取独立 `*sql.DB`（推荐），而非像 SQLite 那样共享。这样每个 Store 可以独立配置连接池大小。

```go
// Phase 6 后的 initStores 伪代码
func initStores(ctx context.Context, cfg *config.Config, log *slog.Logger) (*gatewayStores, error) {
    switch dbutil.ParseDialect(cfg.DB.Driver) {
    case dbutil.DialectSQLite:
        writeMu := sqlutil.NewWriteMu()
        ss, _ := session.NewSQLiteStore(ctx, cfg, writeMu)       // 现有代码路径
        es := eventstore.NewSQLiteStore(ss.DB(), writeMu)
        return &gatewayStores{session: ss, event: es, writeMu: writeMu}, nil
    case dbutil.DialectPostgres:
        ss, _ := session.NewPGStore(ctx, cfg)                     // 新 PG 路径
        es, _ := eventstore.NewPGStore(ctx, cfg)                  // 独立 DB
        return &gatewayStores{session: ss, event: es}, nil        // 无 WriteMu
    }
}
```

## 7. 风险与缓解

| 风险 | 严重度 | 缓解措施 |
|------|--------|----------|
| Rebind 状态机缺陷导致 SQL 注入或语法错误 | 高 | 150 行测试覆盖所有边界；参考 autobrr/qui 生产级实现 |
| PG 迁移与 SQLite 迁移不一致 | 高 | 双驱动迁移测试；逐字段对比 DDL |
| PG Store 中 `ON CONFLICT ... DO UPDATE` 语法细节差异 | 中 | 从 Grafana 的 `UpsertSQL()` 实现中学习；写集成测试 |
| PG 连接池配置不当导致连接耗尽 | 中 | 参考 PostgreSQL 官方最佳实践：MaxOpenConns=25, MaxIdleConns=5, MaxLifetime=5m |
| SQLite 路径退化 | 低 | 所有 SQLite 测试仍必须通过；CI 双矩阵 |
| 性能退化（PG 下 Rebind 每次查询都调用） | 低 | Rebind 是纯字符串操作，纳秒级；可在 Store 构造时预 Rebind 查询 |

## 10. 附录

### A. Rebind 状态机设计

```
States: Normal → SingleQuote → DoubleQuote → DollarQuote → BlockComment → LineComment

Transitions:
Normal + '?'         → emit $N, stay Normal
Normal + '''          → SingleQuote
SingleQuote + '''''   → Normal (escaped quote = two single quotes)
SingleQuote + '''     → Normal (closing quote)
Normal + '"'          → DoubleQuote
DoubleQuote + '""'    → Normal (escaped quote)
DoubleQuote + '"'     → Normal (closing quote)
Normal + '$' + tag    → DollarQuote
DollarQuote + '$' + tag + '$' → Normal
Normal + '/' + '*'    → BlockComment
BlockComment + '*' + '/' → Normal
Normal + '-' + '-'    → LineComment
LineComment + '\n'    → Normal
```
